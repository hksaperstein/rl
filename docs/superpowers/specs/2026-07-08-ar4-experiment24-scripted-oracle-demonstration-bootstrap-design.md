# Experiment 24: Scripted-Oracle Demonstration Bootstrapping

## Scope note

This spec describes both gates so Gate 2's design is grounded and
committed to now, rather than re-derived later — but Gate 2's exact
implementation plan is NOT written until Gate 1's actual results are in
hand (which episodes succeeded, how many, what the recorded
observation/action distributions look like). Per this repo's own
plan-decomposition practice, this becomes two separate implementation
plans executed in sequence: Plan A (Gate 1 only) now; Plan B (Gate 2)
authored only after Plan A's results are known and reviewed. This spec
is not violating the "no placeholders" rule so much as scoping which
parts are plannable now vs. only after a real, load-bearing dependency
resolves.

## Context and framing

Eight consecutive experiments (13, 17-23) spanning reward shaping, grasp
gating, orientation bias, proximity gating, software jaw mirroring, and
warm-started residual RL over a classical controller have all converged
on the identical null result: `lifting_object`/`both_magnitude_ok_steps`
never go nonzero. Every one of these was a variant within the same
technique family — reward/action-space engineering layered on top of
pure PPO exploration. Per this repo's mandate to default toward a
structurally different strategy after a string of nulls, this
experiment is the first attempt at demonstration/imitation bootstrapping
— a technique family never tried in this repo before.

## Hypothesis

**PPO exploration alone has never discovered a coordinated grasp+lift
because the compound action sequence required (approach → align → close
the gripper at the right moment and position → lift) is a low-probability
conjunction under random/entropy-driven exploration, independent of how
well-shaped the reward is — this repo's own history (eight null
experiments across every reward variant tried) is itself evidence
against "the reward just needs to be shaped better" and consistent with
"the exploration problem itself is the bottleneck," which reward shaping
cannot fix but demonstrations directly address by showing the policy at
least one successful trajectory through that low-probability region
before it has to discover one on its own.**

Falsifiable at two gates (see Design):

- **Gate 1 (scripted-oracle viability)**: if a pure, non-learned scripted
  controller — using this repo's own existing waypoint geometry with a
  hand-coded gripper-close schedule, no RL involved — cannot reliably
  grasp+lift the cube either, this falsifies the premise that a
  demonstrable successful trajectory even exists using this repo's
  current waypoint/gripper mechanism, independent of the exploration
  argument entirely. That would mean the bottleneck is not exploration at
  all, but something more fundamental about the action interface, gripper
  mechanics, or waypoint geometry itself — a different, likely
  asset/mechanism-level, problem.
- **Gate 2 (BC-pretrain + RL-finetune)**: given Gate 1 passes, if a
  policy pretrained via behavior cloning on the oracle's demonstrations,
  then fine-tuned with PPO, still cannot grasp+lift, this falsifies the
  exploration-bottleneck hypothesis specifically — demonstrations would
  have removed the discovery problem, so a continued null would mean
  something else (e.g. the BC-to-RL transfer itself, or a genuine reward/
  observation gap) is the real blocker, pointing toward a different
  investigation (e.g. auditing whether BC-cloned actions even survive
  the `RslRlVecEnvWrapper`'s `clip_actions` and action-space scaling
  unchanged).

## Background research

Searched directly via Google Scholar/arXiv this session (not delegated):

- **Rajeswaran et al. 2018, "Learning Complex Dexterous Manipulation with
  Deep Reinforcement Learning and Demonstrations" (arXiv:1709.10087)** —
  introduces DAPG (Demonstration Augmented Policy Gradient): a policy-
  gradient RL method that mixes a demonstration-imitation loss into the
  RL objective with a decaying weight, specifically designed for
  exploration difficulty in high-dimensional continuous-control
  manipulation. Directly on-point for this repo's exact failure mode
  (compound manipulation action never discovered by RL alone).
- **Nair et al. 2018, "Overcoming Exploration in Reinforcement Learning
  with Demonstrations" (arXiv:1709.10089, ICRA)** — demonstrates that a
  small set of demonstrations, combined with RL (DDPG+HER), solves
  long-horizon sparse-reward block-stacking tasks that RL alone could
  not — the closest published analogue to this repo's own task
  structure (sparse/gated grasp+lift+place reward, continuous control).
- **"Learning visual servo policies via planner cloning"
  (arXiv:2005.11810)** and the broader planner-cloning/TAMP-plus-RL
  literature — establishes that demonstrations generated by a scripted
  planner/controller (not human teleoperation) are a valid, published
  substitute for human demos, directly resolving this repo's own
  previously-identified blocker (Isaac Lab Mimic requiring human
  teleoperation, ~10 demos minimum, not autonomously executable).
- **TMRL (arXiv:2605.12236) and Posterior Behavioral Cloning
  (arXiv:2512.16911)**, both 2025-2026 work — flag a specific, concrete
  pitfall this design must account for: BC pretraining produces a
  narrow action distribution that creates an exploration bottleneck for
  RL fine-tuning, and naive fine-tuning with an untrained/randomly-
  initialized critic can catastrophically forget the BC-learned
  behavior before the critic has learned enough to guide useful
  updates.

This last pitfall has a direct, already-validated mitigation available
in this repo's own recent history: Experiment 23's warm-start ramp
(residual authority held near zero while the value function gets real
training signal first, per Johannink et al. 2019) is structurally the
same idea Johannink applied to residual RL — hold back an
undertrained component's influence until a more-trained component (the
critic) can support it. Experiment 23 independently verified (Task 4's
hard gate, reviewed and confirmed) that this kind of ramp mechanism can
be implemented and actually works as designed in this exact codebase.
Gate 2's design reuses that same warm-start principle, adapted to the
BC-to-PPO transfer instead of the residual-authority transfer.

## Design

### Gate 1: Scripted-oracle viability (must pass before Gate 2 is built)

**New standalone script**, `scripts/oracle_rollout.py` — no RL, no
policy network, no `ActionTerm`/env-cfg action wiring at all. A single
Python loop that:

1. Constructs `Ar4PickPlaceMirrorSceneCfg`-based env (reusing the
   existing scene — cube spawn randomization, mirrored goal, contact
   sensors — unchanged from every experiment this session).
2. Registers `compute_path_waypoints` as a reset event (already proven
   compatible with this scene in Experiments 13/23) to get the 5
   geometric waypoints (pregrasp/grasp/lift/transit/place) each episode.
3. Each step: reads the currently-active waypoint
   (`env._path_waypoint_idx`), computes a live differential-IK joint
   target toward it — reusing `ik_guided_path_bonus`'s exact established
   IK-controller setup pattern (`DifferentialIKController`,
   `command_type="position"`, `use_relative_mode=False`, `ik_method="dls"`,
   plus the pinch-point offset correction already documented and
   working in `tasks/ar4/mdp.py`) — and commands the robot's arm joints
   directly to that target via `robot.set_joint_position_target()`.
   Advances the waypoint index using the same `advance_tolerance`
   distance check already used elsewhere in this repo (0.03m).
4. Gripper: a **hand-coded schedule tied to waypoint index** — open
   (`GRIPPER_OPEN_POS`) at waypoints 0 (pregrasp) and 4 (place), closed
   (`GRIPPER_CLOSED_POS`) at waypoints 1-3 (grasp/lift/transit). This is
   the simplest possible non-learned gripper policy: close once
   positioned at the grasp waypoint, stay closed through lift and
   transit, release at place. No proximity gating, no force feedback —
   deliberately as simple as possible, since the goal is to establish
   *a* working baseline, not the best possible one.
5. Records the full per-step `(observation, action)` trajectory (in the
   exact same observation/action space `Ar4PickPlaceWarmResidualEnvCfg`
   or a plain task-space equivalent uses, so Gate 2 can consume it
   directly without a format conversion step) to disk (e.g. an
   `.npz`/`.pt` file per episode) for every episode, tagged with whether
   that episode's rollout satisfied the grasp+lift success criteria.

**Verification**: run for at least 50 episodes (cheap — no training, no
GPU-heavy PPO loop, just physics stepping). Reuse the exact same
instrumented contact-diagnostic criteria this session has used
throughout (`force_threshold=0.05`, `antipodal_cos_threshold=-0.7071`,
`minimal_height=0.03`, `both_magnitude_ok`/`antipodal_ok`/`height_ok`/
`gate_fires`) to score each episode. Success = the same
`genuine_grasp_and_lift` gate this repo's own reward functions use,
verified via video review of at least 3 successful episodes before
trusting the oracle's own self-reported success (per this project's own
Experiment-16-established standard: a scalar/counter claiming success is
not sufficient without visual confirmation the object is genuinely held
between the jaws, not wedged against the wrist).

**Hard gate**: if fewer than ~30/50 episodes (60%) pass the strict
grasp+lift gate, or if video review of the "successful" episodes reveals
wedging/false positives rather than genuine grasps, Gate 1 fails. Report
this plainly — do not proceed to Gate 2's implementation, and do not
attempt further oracle tuning beyond one bounded fix cycle (per this
repo's systematic-debugging discipline: if the naive hand-coded
gripper schedule doesn't work, try ONE specific, literature/geometry-
grounded refinement — e.g. tightening `advance_tolerance` specifically
at the grasp waypoint so the arm is more precisely positioned before
closing — not open-ended parameter search).

### Gate 2: BC pretrain + warm-started RL finetune (conditional on Gate 1)

**Approach chosen: two-stage BC pretrain → warm-started PPO finetune**,
over the alternatives considered:

- *DAPG-style continuous demo-loss mixing* (Rajeswaran et al.'s own
  published design) was considered but requires modifying `rsl_rl`'s
  `OnPolicyRunner`/PPO update loop internals to inject an auxiliary
  imitation loss term throughout training — a third-party library this
  repo has so far only ever configured, never forked/modified
  internally. Real, concrete extra implementation risk for uncertain
  benefit over the simpler two-stage approach, given this repo's actual
  failure history has never shown a *catastrophic-forgetting*-style
  problem (the risk DAPG's continuous mixing specifically protects
  against) — it has shown a pure *discovery* problem, which a one-time
  BC pretrain already directly addresses.
- *Demo-initialized episode resets only* (state-only curriculum,
  no BC loss at all) was considered but discarded: this repo's own
  Experiments 18/21 already gave the policy dense proximity/pregrasp-
  readiness shaping specifically rewarding being *near* a good grasp
  state — and grasp discovery still never happened. Starting episodes
  from near-grasp states without also demonstrating the *actions* that
  close the gripper correctly likely reproduces the same gap, since the
  missing ingredient across all 8 prior nulls was never "how do I get
  close," it was consistently "what do I do once I'm close."

**BC pretraining** (new script, `scripts/oracle_bc_pretrain.py`):
supervised regression of the PPO actor network (same architecture as
`Ar4PickPlacePPORunnerCfg`'s `actor_hidden_dims=[256, 128, 64]`,
`activation="elu"`) against the oracle's recorded `(observation, action)`
pairs from Gate 1's successful episodes only (MSE loss on continuous
action outputs), trained to convergence (loss plateau, not a fixed
epoch count) on a held-out validation split of the recorded episodes.

**Warm-started RL finetune**: a new PPO run initialized from the
BC-pretrained actor's weights, with the CRITIC trained on the BC
policy's own on-policy rollout returns for an initial window (mirroring
Johannink et al.'s and Experiment 23's own warm-start technique: let the
critic catch up to what the actor already knows how to do before large
policy-gradient updates are allowed to move the actor away from the BC
solution). Concretely: for the first `N` iterations (a bounded, small
fraction of the full 1500-iteration budget — a specific value to be
set in the implementation plan after checking how many iterations a
value-function-only warm-up realistically needs, following this
project's own precedent of deriving warmup duration from a concrete
calculation, not a round-number guess), scale down the policy-loss
gradient's contribution (or equivalently, hold `entropy_coef`/
`clip_param` conservative) while the value-function loss trains
normally — then transition to standard full PPO for the remainder.

**Reward/observation/environment**: reuses Experiment 22's exact
reward/observation lineage unchanged (the current best-performing
configuration), matching this session's established "one new variable
at a time" discipline — the only new variables in Gate 2 are (a) the
BC-pretrained initial actor weights and (b) the warm-started critic
finetune schedule, isolated from any reward-function change.

## What this does NOT change

No modification to any existing reward function, any existing env cfg,
`compute_path_waypoints`, or any prior action term. Purely additive: two
new standalone scripts (`oracle_rollout.py`, `oracle_bc_pretrain.py`)
plus one new RL-finetune entry point/env-cfg variant for Gate 2, built
only if Gate 1 passes.

## Verification plan

Gate 1: 50-episode scripted rollout, scored via the established contact-
diagnostic criteria, video-verified on a sample of "successful"
episodes before trusting the oracle. Hard gate — do not proceed to Gate
2's implementation without this passing.

Gate 2 (conditional): BC pretraining loss curve (train/val split) to
confirm real learning, not memorization/collapse; a warm-start
verification analogous to Experiment 23's Task 4 (confirm the critic-
warm-up schedule genuinely delays large policy updates, not just
asserted); a 300-iteration diagnostic run; the full 1500-iteration run;
the same instrumented contact diagnostic used throughout this session;
video review before any success claim, per this project's standing
practice.

## Success criteria

Primary: does Gate 1's scripted oracle achieve a genuine (video-
verified) grasp+lift at a meaningfully-above-chance rate. If yes, does
Gate 2's BC-pretrained-and-finetuned policy's `lifting_object`/
`both_magnitude_ok_steps` move off the current unbroken `0` streak. A
Gate 2 null, with Gate 1 passing and the warm-start mechanism
independently verified working (matching Experiment 23's own
verification standard), would specifically falsify the exploration-
bottleneck hypothesis and motivate investigating the BC-to-RL transfer
mechanism itself as the next concrete question.
